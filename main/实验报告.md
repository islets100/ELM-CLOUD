# 基于Spring Cloud微服务架构的外卖点餐平台实验报告

---

## 一、需求分析

### 1.1 项目背景

本项目是"饿了么"外卖点餐平台的微服务化实现，基于 Spring Cloud + Vue 前后端分离架构。系统围绕外卖点餐的核心业务流程，涵盖用户注册登录、商家浏览、食品选择、购物车管理、订单创建与支付、虚拟钱包、积分系统等功能模块。

### 1.2 功能性需求

#### 1.2.1 用户服务模块
- 用户注册：支持用户名、密码、性别等基本信息的注册
- 用户登录：基于JWT（JSON Web Token）的身份认证，支持记住我功能
- 用户信息管理：查看个人信息、修改密码、更新用户资料
- 生日设置：用于生日积分奖励的判断依据

#### 1.2.2 商家服务模块
- 商家列表查询：按分类展示所有商家
- 商家详情查询：获取特定商家的完整信息（名称、地址、起送价、配送费等）
- 商家注册与管理：支持商家入驻和管理自己的店铺
- 订单类型管理：按商家分类管理订单类型

#### 1.2.3 食品服务模块
- 食品CRUD：食品的创建、查询、更新、删除（逻辑删除）
- 按商家查询食品：获取某商家下的所有在售食品

#### 1.2.4 购物车服务模块
- 购物车管理：添加食品到购物车、更新数量、删除购物车项
- 按用户和商家维度查询购物车：支持分别按用户和商家查询购物车内容
- 订单创建联动：下单后自动清空对应购物车

#### 1.2.5 订单服务模块
- 订单创建：从购物车创建订单，包含订单明细
- 订单查询：按用户、商家、订单ID多维度查询
- 订单状态流转：待支付→已支付→配送中→已完成
- 虚拟钱包支付：通过虚拟钱包余额+积分抵扣完成支付
- 分布式锁保障支付一致性

#### 1.2.6 配送地址服务模块
- 配送地址CRUD：新增、修改、删除配送地址
- 按用户查询所有有效地址

#### 1.2.7 虚拟钱包服务模块
- 钱包自动创建：用户注册后自动创建虚拟钱包
- 余额查询、充值、支付、转账
- 透支管理：支持信用额度透支与还款
- 交易明细记录

#### 1.2.8 VIP卡服务模块
- VIP卡购买：支持99元和299元两档VIP卡
- 信用额度：VIP用户获得2000元信用额度
- VIP卡状态管理

#### 1.2.9 积分服务模块
- 积分获取：签到、消费返积分、活动赠送、生日奖励等
- 积分消费：订单抵扣、积分兑换
- 积分统计：按可用/已用/已过期维度统计
- 定时任务：每日凌晨积分过期处理、生日积分自动发放
- 红包雨活动

### 1.3 非功能性需求

#### 1.3.1 性能需求
- 网关层引入响应缓存机制，对商家和食品等低频变更数据的GET请求进行缓存，减少重复的微服务调用
- 数据库连接池复用，减少连接创建开销
- 使用消息队列实现异步通信，提升系统吞吐量

#### 1.3.2 安全需求
- JWT令牌认证，所有API请求需携带有效Token
- BCrypt密码加密存储
- 网关统一入口，隐藏内部微服务拓扑

#### 1.3.3 可用性需求
- 服务注册与发现机制，支持服务实例动态上下线
- 网关层熔断降级，保证每个请求都有响应（即使后端服务故障）
- Feign客户端集成降级回退逻辑

#### 1.3.4 可维护性
- 集中配置管理，所有服务配置统一托管在Config Server
- 基于Spring Cloud Bus的配置动态刷新
- RESTful API规范化设计，前后端通过接口规格说明书解耦开发

#### 1.3.5 可扩展性
- 微服务独立部署、独立扩展
- 服务间通过Feign声明式调用，无硬编码依赖
- 数据库按业务域垂直拆分

#### 1.3.6 一致性
- 钱包、积分、订单等涉及金钱的业务使用Redisson分布式锁保障严格一致性
- 订单支付流程使用事务保证数据一致性

## 三、详细设计

### 3.1 基于Eureka的服务注册与发现

#### 3.1.1 设计原理

Eureka是Netflix开源的服务注册与发现组件，采用AP架构（可用性+分区容错性），包含两个核心角色：

- **Eureka Server（服务注册中心）**：维护所有已注册服务的实例列表，提供服务注册表
- **Eureka Client（服务客户端）**：启动时向Server注册，定期发送心跳续约，从Server拉取服务列表

#### 3.1.2 本项目实现

**Eureka Server配置**（[elm-eureka-server](elm-eureka-server)）：

```yaml
server:
  port: 13000
eureka:
  instance:
    hostname: 127.0.0.1
  client:
    register-with-eureka: false   # 不注册自身
    fetch-registry: false         # 不拉取注册表
    service-url:
      defaultZone: http://${eurekahost2}:${server.port}/eureka/
```

关键设计点：
- `register-with-eureka: false` 和 `fetch-registry: false`：因为当前节点本身就是Server，不需要向自己注册
- `defaultZone` 配置了同伴节点地址，为Eureka集群做准备

**Eureka Client配置**（各业务微服务的bootstrap.yml）：

```yaml
eureka:
  client:
    service-url:
      defaultZone: http://127.0.0.1:13000/eureka/
  instance:
    prefer-ip-address: true
    lease-renewal-interval-in-seconds: 5      # 心跳间隔5秒
    lease-expiration-duration-in-seconds: 15  # 过期时间15秒
```

所有微服务通过 `@EnableEurekaClient`（实际由 `@EnableDiscoveryClient` 提供能力）注解启用服务发现，启动后自动注册到Eureka Server。

### 3.2 Eureka高可用集群设计

#### 3.2.1 设计原理

单点Eureka Server存在单点故障风险。Eureka通过Peer-to-Peer（对等复制）机制实现高可用：部署多个Eureka Server实例，它们互相注册，服务注册信息在集群间同步。

#### 3.2.2 本项目实现

项目配置了两个Eureka节点（eurekahost1 和 eurekahost2），通过 `defaultZone` 互相指向对方实现集群：

```
┌──────────────────┐         注册对方          ┌──────────────────┐
│  Eureka Server 1 │ ◄───────────────────────── │  Eureka Server 2 │
│  host1:13000     │ ─────────────────────────► │  host2:13000     │
└──────────────────┘                            └──────────────────┘
        ▲                                               ▲
        │ 注册                                          │ 注册
   ┌────┴────┐                                    ┌────┴────┐
   │ 微服务A  │                                    │ 微服务B  │
   └─────────┘                                    └─────────┘
```

各微服务配置了两个注册中心地址：
```yaml
eureka:
  client:
    service-url:
      defaultZone: http://host1:13000/eureka/,http://host2:13000/eureka/
```

即使一个Eureka Server宕机，服务仍可从另一个Server获取注册表，保证了服务发现的高可用性。

### 3.3 基于Http的客户端负载均衡

#### 3.3.1 设计原理

在微服务架构中，一个服务通常有多个实例。客户端负载均衡在服务调用方实现，调用方从服务注册中心获取目标服务的所有实例列表，然后根据负载均衡算法选择一个实例进行调用。

Spring Cloud 2020.0.2中，Netflix Ribbon已进入维护模式，被Spring Cloud LoadBalancer取代。两者的核心思想一致：在客户端实现负载均衡策略，而非依赖服务端或硬件负载均衡器。

#### 3.3.2 本项目实现

**Gateway路由中的负载均衡**（[elm-gateway-server-1-dev.yml](elm-configs/elm-gateway-server-1-dev.yml)）：

所有路由均使用 `lb://` 前缀指定目标服务：
```yaml
routes:
  - id: userServer
    uri: lb://elm-user-server    # lb:// 表示启用负载均衡
```

Gateway在转发请求时，通过 `lb://` 前缀触发负载均衡机制，从Eureka注册表中获取 `elm-user-server` 的所有健康实例，然后使用轮询（Round Robin）策略选择一个实例。

**Feign客户端的负载均衡**：

各微服务使用的 `@FeignClient` 注解也是通过 `lb://` 机制实现负载均衡。例如：

```java
@FeignClient(name = "elm-business-server")
public interface BusinessClient {
    @GetMapping("/businesses/{businessId}")
    String getBusinessById(@PathVariable("businessId") Integer businessId);
}
```

Feign内部集成了Spring Cloud LoadBalancer，当调用 `elm-business-server` 时，自动从Eureka获取该服务的实例列表并负载均衡。

**Eager Load配置**：Gateway配置了急切加载，避免首次请求延迟：
```yaml
spring:
  cloud:
    gateway:
      loadbalancer:
        eager-load:
          clients:
            - elm-user-server
            - elm-business-server
            # ...所有8个业务服务
```

### 3.4 基于Feign的声明式服务调用

#### 3.4.1 设计原理

Feign是Netflix开发的声明式HTTP客户端，通过接口+注解的方式定义服务调用契约，简化了服务间通信代码。相比直接使用RestTemplate，Feign的优势在于：

1. **声明式编程模型**：只需定义接口和注解，无需编写HTTP调用样板代码
2. **内置负载均衡**：集成Spring Cloud LoadBalancer，自动实现客户端负载均衡
3. **集成熔断降级**：通过fallback机制实现服务降级
4. **契约驱动**：接口定义即为服务契约，便于前后端和跨团队协作

#### 3.4.2 本项目实现

项目中共定义了以下Feign客户端：

| Feign客户端 | 所在服务 | 目标服务 | 用途 |
|------------|---------|---------|------|
| BusinessClient | food-server | elm-business-server | 验证商家是否存在 |
| FoodFeignClient | cart-server | elm-food-server | 获取食品信息 |
| OrderFeignClient | cart-server | elm-business-server | 创建订单 |
| BusinessClient | order-server | elm-business-server | 获取商家信息 |
| CartClient | order-server | elm-cart-server | 购物车操作 |
| PointClient | order-server | elm-point-server | 积分操作 |
| VirtualWalletClient | order-server | elm-wallet-server | 钱包支付 |
| UserClient | wallet-server | elm-user-server | 验证用户存在 |
| OrderClient | point-server | elm-order-server | 订单积分关联 |

**示例：订单服务调用钱包服务**（[VirtualWalletClient.java](elm-order-server/src/main/java/team/tjusw/elm/feign/VirtualWalletClient.java)）：

```java
@FeignClient(name = "elm-wallet-server")
public interface VirtualWalletClient {
    @GetMapping("/wallet/getBalance")
    CommonResult<BigDecimal> getBalanceByUserId(@RequestParam("userId") String userId);

    @PostMapping("/wallet/transfer")
    CommonResult<Integer> transfer(@RequestParam("fromUserId") String fromUserId,
                                    @RequestParam("toUserId") String toUserId,
                                    @RequestParam("amount") BigDecimal amount);
}
```

**Feign集成Fallback降级**（[BusinessClientFallback.java](elm-food-server/src/main/java/team/tjusw/elm/feign/BusinessClientFallback.java)）：

```java
@FeignClient(name = "elm-business-server", fallback = BusinessClientFallback.class)
public interface BusinessClient {
    @GetMapping("/businesses/{businessId}")
    String getBusinessById(@PathVariable("businessId") Integer businessId);
}

@Component
public class BusinessClientFallback implements BusinessClient {
    @Override
    public String getBusinessById(Integer businessId) {
        return "{\"code\":200, \"message\":\"Fallback: Business service unavailable\"}";
    }
}
```

当 `elm-business-server` 不可用时，Feign自动降级调用 `BusinessClientFallback`，返回模拟的成功响应，防止食品服务因商家服务不可用而完全失败。

### 3.5 基于Resilience4j的延迟与容错

#### 3.5.1 设计原理

Hystrix是Netflix开源的传统熔断器组件，但在Spring Cloud 2020.0.2中已被移除，本项目使用Resilience4j作为替代方案，应用于Gateway网关层实现熔断降级。

**熔断器（Circuit Breaker）三态模型**：

```
       ┌─────────┐
       │ CLOSED  │  ← 正常状态，请求正常通过
       └────┬─────┘
            │ 达到失败阈值
            ▼
       ┌─────────┐
       │  OPEN   │  ← 熔断状态，请求直接失败/降级
       └────┬─────┘
            │ 等待超时后进入半开
            ▼
       ┌──────────┐
       │ HALF_OPEN│  ← 试探性放行部分请求
       └────┬──────┘
            │ 试探成功→CLOSED / 试探失败→OPEN
```

#### 3.5.2 本项目实现

**熔断器配置**（[CircuitBreakerConfig.java](elm-gateway-server/src/main/java/team/tjusw/config/CircuitBreakerConfig.java)）：

```java
@Configuration
public class CircuitBreakerConfig {
    @Bean
    public ReactiveResilience4JCircuitBreakerFactory reactiveResilience4JCircuitBreakerFactory() {
        ReactiveResilience4JCircuitBreakerFactory factory = 
            new ReactiveResilience4JCircuitBreakerFactory();
        TimeLimiterConfig timeLimiterConfig = TimeLimiterConfig.custom()
                .timeoutDuration(Duration.ofSeconds(20))  // 20秒超时
                .cancelRunningFuture(true)                // 超时取消运行中的Future
                .build();
        factory.configure(builder -> builder.timeLimiterConfig(timeLimiterConfig).build(),
                "globalCircuitBreaker",
                "userCircuitBreaker", "businessCircuitBreaker",
                "foodCircuitBreaker", "cartCircuitBreaker",
                "orderCircuitBreaker", "deliveryCircuitBreaker",
                "walletCircuitBreaker", "pointCircuitBreaker");
        return factory;
    }
}
```

系统为9个服务分别创建了独立的熔断器，每个超时时间20秒。这样的隔离设计确保某个服务熔断不会影响其他服务的正常访问。

**路由中的熔断器配置**：

Gateway的15条路由规则中，每条都配置了CircuitBreaker过滤器和对应的fallback地址：

```yaml
routes:
  - id: userServer
    uri: lb://elm-user-server
    predicates:
      - Path=/users/**
    filters:
      - name: CircuitBreaker
        args:
          name: userCircuitBreaker
          fallbackUri: forward:/userFallback
```

**降级响应设计**（[FallbackController.java](elm-gateway-server/src/main/java/team/tjusw/controller/FallbackController.java)）：

```java
@RestController
public class FallbackController {
    @RequestMapping("/userFallback")
    @ResponseStatus(HttpStatus.FORBIDDEN)
    public CommonResult<Map<String, Object>> userFallback(ServerWebExchange exchange) {
        Map<String, Object> metadata = new LinkedHashMap<>();
        metadata.put("fallback", true);           // 标记为熔断响应
        metadata.put("service", "user");          // 标识故障服务
        metadata.put("retryable", true);          // 标记可重试
        metadata.put("path", exchange.getRequest().getURI().getRawPath());
        metadata.put("method", exchange.getRequest().getMethodValue());
        return new CommonResult<>(403, "Gateway触发了用户微服务的熔断降级方法.", metadata);
    }
}
```

降级响应包含 `fallback: true` 和 `retryable: true` 元数据，前端可据此展示友好提示并引导用户重试。

### 3.6 基于Spring Cloud Gateway的微服务网关

#### 3.6.1 设计原理

Spring Cloud Gateway是基于Spring WebFlux的响应式API网关，核心组件包括：

- **Route（路由）**：路由是网关的基本构建块，包含ID、目标URI、断言集合和过滤器集合
- **Predicate（断言）**：匹配HTTP请求的条件（路径、Header、参数等）
- **Filter（过滤器）**：对请求/响应进行修改处理，分为GatewayFilter（路由级）和GlobalFilter（全局级）

#### 3.6.2 本项目实现

**全局CORS配置**：

```yaml
spring:
  cloud:
    gateway:
      globalcors:
        cors-configurations:
          '[/**]':
            allowedOrigins: "*"
            allowedHeaders: "*"
            allowedMethods: "*"
```

**HTTP超时配置**：

```yaml
spring:
  cloud:
    gateway:
      httpclient:
        connect-timeout: 20000    # 连接超时20秒
        response-timeout: 20s     # 响应超时20秒
```

**路由设计**：共配置15条路由，每个业务微服务有两条路由——直接路径路由和带 `/api/` 前缀路由：

| 路由ID | 路径匹配 | 目标服务 | 特殊过滤器 |
|--------|---------|---------|-----------|
| userServer | /users/** | elm-user-server | CircuitBreaker |
| userApiServer | /api/auth, /api/register, ... | elm-user-server | CircuitBreaker |
| businessServer | /businesses/** | elm-business-server | CircuitBreaker, BusinessOrderTypeCacheGet, BusinessCacheGet |
| businessApiServer | /api/businesses/** | elm-business-server | RewritePath, CircuitBreaker, 缓存过滤器 |
| cartServer | /carts/** | elm-cart-server | CircuitBreaker |
| ... | ... | ... | ... |

**路径重写过滤器**：`/api/` 前缀路由使用 `RewritePath` 过滤掉 `/api` 前缀：
```yaml
filters:
  - RewritePath=/api/(?<segment>.*), /${segment}
```

这样前端调用 `http://localhost:14000/api/businesses/1`，网关将其重写为 `/businesses/1` 后转发到 `elm-business-server`。

#### 3.6.3 自定义缓存过滤器（项目特色功能）

针对商家和食品数据变更频率低、读取频率高的特点，在网关层实现了自定义缓存机制：

**三类缓存过滤器**：

1. **BusinessCacheGet**：对 `GET /businesses/{id}` 请求，在转发前检查本地缓存。若命中缓存，直接返回缓存内容，不再调用后端服务
2. **BusinessOrderTypeCacheGet**：缓存商家订单类型数据
3. **BusinessFoodCacheGet**：缓存商家食品列表

**缓存实现**（[BusinessCacheGetGatewayFilterFactory.java](elm-gateway-server/src/main/java/team/tjusw/filter/BusinessCacheGetGatewayFilterFactory.java)）：

```java
@Component
public class BusinessCacheGetGatewayFilterFactory 
        extends AbstractGatewayFilterFactory<Object> {
    
    private static final String PATH_PATTERN = "businesses/(\\d+)";

    @Override
    public GatewayFilter apply(Object config) {
        return (exchange, chain) -> {
            String path = exchange.getRequest().getURI().getPath();
            // 匹配资源ID
            Matcher matcher = Pattern.compile(PATH_PATTERN).matcher(path);
            if (matcher.find()) {
                Integer businessId = Integer.valueOf(matcher.group(1));
                // 缓存命中，直接返回
                if (BusinessCache.businessMap.containsKey(businessId)) {
                    return exchange.getResponse().writeWith(
                        Mono.just(exchange.getResponse().bufferFactory().wrap(
                            BusinessCache.businessMap.get(businessId).getBytes()
                        ))
                    );
                }
            }
            return chain.filter(exchange);  // 未命中，继续转发
        };
    }
}
```

**缓存策略总结**：

| 请求类型 | 缓存行为 |
|---------|---------|
| GET 单个商家/食品 | 缓存未命中→转发后端→缓存响应；缓存命中→直接返回 |
| PUT/POST/DELETE | 清除相关缓存，确保数据一致性 |

此设计将重复请求的响应时间从秒级降至毫秒级，显著提升了用户体验。

### 3.7 基于Spring Cloud Config的集中配置管理

#### 3.7.1 设计原理

在微服务架构中，每个服务都有自己的配置文件（application.yml）。当服务数量增多时，分散的配置文件难以管理和维护。Spring Cloud Config提供了服务端-客户端模式的集中配置方案：

- **Config Server**：从Git仓库或本地文件系统读取配置文件，通过REST API对外暴露
- **Config Client**：启动时从Config Server拉取配置，再初始化自己的Spring上下文

**Config Server配置**（[elm-config-server](elm-config-server/src/main/resources/application.yml)）：

```yaml
  cloud:
    config:
      server:
        native:
          search-locations: file:///E:/ELM-CLOUD/elm-configs  # 配置文件存储路径
```

**配置文件命名规范**：`{application-name}-{profile}.yml`

### 3.8 基于Spring Cloud Bus的配置刷新

#### 3.8.1 设计原理

当Config Server中的配置发生变化时，需要通知各微服务刷新配置。Spring Cloud Bus利用消息中间件（如RabbitMQ）作为通信总线，实现配置变更的广播通知：

```
Config Server ──(POST /actuator/busrefresh)──> RabbitMQ
                                                    │
                           ┌────────────────────────┼────────────────────────┐
                           ▼                        ▼                        ▼
                      Service A               Service B               Service C
                   (刷新配置)               (刷新配置)               (刷新配置)
```

#### 3.8.2 本项目实现

**Config Server暴露刷新端点**：

```yaml
management:
  endpoints:
    web:
      exposure:
        include:
          - busrefresh    # 暴露 /actuator/busrefresh 端点
```

**各业务微服务依赖**：

```xml
<dependency>
    <groupId>org.springframework.cloud</groupId>
    <artifactId>spring-cloud-bus</artifactId>
</dependency>
<dependency>
    <groupId>org.springframework.cloud</groupId>
    <artifactId>spring-cloud-stream-binder-rabbit</artifactId>
</dependency>
```

**配置刷新流程**：

1. 管理员修改Config Server中配置文件的内容
2. 发送POST请求到 `http://config-server:15001/actuator/busrefresh`
3. Config Server通过RabbitMQ广播 `RefreshRemoteApplicationEvent` 事件
4. 所有订阅了该事件的微服务收到通知
5. 各微服务重新从Config Server拉取最新配置
6. 标注了 `@RefreshScope` 的Bean自动重建

这种机制避免了对每个微服务逐一发送刷新请求的繁琐操作，只需一次触发即可刷新所有服务。

### 3.9 业务流程设计

#### 3.9.1 用户注册与登录流程

```
用户 → 前端(Register.vue) → Gateway → elm-user-server

注册流程:
1. 前端提交用户名、密码、性别等信息
2. elm-user-server使用BCrypt加密密码
3. 写入MySQL user表，初始delTag=1（正常状态）

登录流程:
1. 前端提交用户名、密码
2. elm-user-server查询user表验证用户
3. 验证密码（支持BCrypt和明文两阶段兼容，明文密码首次登录后自动升级为BCrypt）
4. 生成JWT Token（HMAC-SHA512签名），包含userId和authorities
5. 返回Token给前端，前端存储到localStorage
6. 后续请求在Header中携带: Authorization: Bearer {token}
```

#### 3.9.2 点餐与下单流程

```
前端 → Gateway → 各业务微服务

1. 浏览商家: GET /api/businesses → elm-business-server
   网关缓存过滤器介入，命中缓存直接返回

2. 浏览食品: GET /api/foods?businessId={id} → elm-food-server
   Feign调用 elm-business-server 验证商家存在
   网关缓存过滤器介入

3. 加入购物车: POST /api/carts → elm-cart-server
   Feign调用 elm-food-server 验证食品存在和所属商家

4. 创建订单: POST /api/orders → elm-order-server
   ① Feign获取购物车内容 (CartClient → elm-cart-server)
   ② 创建Orders主记录 (写入orders表)
   ③ 批量创建订单明细 (写入orderDetailet表)
   ④ Feign清空购物车 (CartClient → elm-cart-server)
   ⑤ 全程在 @Transactional 事务保护下
```

#### 3.9.3 支付流程（虚拟钱包+积分）

```
POST /api/orders/{orderId}/pay/virtual-wallet → elm-order-server

支付流程 (processPayment):
1. 获取分布式锁 Redisson (key="order-{orderId}", 超时=10s)
2. 检查订单状态，已支付则返回3
3. Feign获取商家userId (BusinessClient → elm-business-server)
4. Feign查询用户钱包余额 (VirtualWalletClient → elm-wallet-server)
5. Feign查询用户积分余额 (PointClient → elm-point-server)
6. 余额不足 → 返回1 | 积分不足 → 返回2
7. Feign计算积分抵扣金额 (PointClient → elm-point-server)
8. 实际支付金额 = 订单总额 - 积分抵扣金额
9. Feign转账 fromUserId→businessUserId (VirtualWalletClient → elm-wallet-server)
10. 转账成功:
    ① Feign发放消费积分 (PointClient → elm-point-server)
    ② 更新订单状态为"已支付" (orderState=1)
    ③ Feign扣除使用积分 (PointClient → elm-point-server)
11. 释放分布式锁
12. 返回支付结果
```

#### 3.9.4 订单状态异步通知流程

```
elm-business-server (订单状态变更)
    │
    │ RabbitMQSender.sendOrderStatusChangeMessage(order)
    ▼
RabbitMQ: order.exchange
    │ routingKey: order.status.change
    ▼
RabbitMQ: order.status.queue
    │
    │ @RabbitListener
    ▼
elm-cart-server (RabbitMQListener)
    │ 处理订单状态变更的业务逻辑
    │ （如清理购物车、发送通知等）
```

#### 3.9.5 积分签到流程

```
POST /api/integral/signIn → elm-point-server

1. 查询用户今日是否已签到 (sign_in_records表)
2. 查询昨日签到记录，计算连续签到天数
3. 计算奖励积分（连续天数越多，奖励越多）
4. 创建签到记录 (sign_in_records表)
5. 写入积分记录 (integral表)
   积分类型: CHECK_IN, 状态: AVAILABLE
6. 返回连续签到天数和获得积分
```

## 四、关键技术点总结

### 4.1 服务间通信方式对比

| 通信方式 | 场景 | 特点 |
|---------|------|------|
| Feign同步调用 | 订单创建、支付（需即时响应） | 声明式、负载均衡、熔断降级 |
| RabbitMQ异步消息 | 订单状态变更通知 | 解耦、异步、可靠投递 |

### 4.2 容错机制分层

```
用户请求
   │
   ▼
Gateway熔断（Resilience4j, Timeout=20s）
   │
   ├── 正常：转发到业务微服务
   └── 熔断：返回FallbackController降级响应
        │
        ▼
Feign调用熔断（Feign Fallback）
   │
   ├── 正常：调用目标微服务
   └── 熔断：执行Fallback实现类
```

### 4.3 数据一致性保障

- **强一致性场景**（支付、转账）：使用Redisson分布式锁，锁超时10秒，防止并发操作
- **最终一致性场景**（订单状态通知）：使用RabbitMQ消息队列异步通知
- **幂等性**：订单支付通过订单状态检查（已支付则返回3）保证幂等性

### 4.4 RESTful API设计原则

项目严格遵循RESTful规范：

- 将数据库表项映射为"资源"，URL表示资源路径
- 使用HTTP方法语义：GET（获取）、POST（创建）、PUT（更新）、DELETE（删除）
- 资源路径使用复数名词：`/businesses`、`/foods`、`/orders`
- 资源层级通过路径参数表达：`/businesses/{businessId}/foods`
- 统一响应格式：`CommonResult(code, message, result)`
- 编写《RESTful API接口规格说明书》，前后端基于文档并行开发

### 4.5 业务对象设计优化

- **金额类型**：全部使用 `BigDecimal`，避免浮点数精度问题
- **对象关联**：业务对象间通过主键关联，不直接嵌套对象。例如Order存储businessId而非Business对象，Cart存储foodId而非Food对象。此设计降低了微服务耦合度，将订单响应大小从几MB降至几KB

## 六、总结

本项目实现了一个基于Spring Cloud微服务架构的外卖点餐平台，完整覆盖了从用户注册、商家浏览、点餐下单到钱包支付、积分管理的全业务流程。项目在以下方面体现了微服务架构的核心能力：

1. **服务治理**：通过Eureka实现服务注册发现，支持集群高可用
2. **负载均衡**：通过Spring Cloud LoadBalancer实现客户端负载均衡
3. **声明式调用**：通过Feign简化了微服务间通信
4. **容错降级**：通过Resilience4j在Gateway层实现熔断，Feign集成Fallback实现降级
5. **API网关**：通过Spring Cloud Gateway实现统一入口、路由转发、自定义缓存
6. **配置管理**：通过Spring Cloud Config实现集中配置，通过Spring Cloud Bus实现配置动态刷新
7. **数据一致性**：通过Redisson分布式锁保障金钱交易的一致性
8. **异步解耦**：通过RabbitMQ实现订单状态异步通知
